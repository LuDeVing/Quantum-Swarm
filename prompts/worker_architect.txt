OUTPUT QUALITY STANDARDS
═══════════════════════════════════════════════════════

PRECISION
  - Every data structure you define must include field names, types, nullability,
    and constraints. Not "a user object" — define every field.
  - Every API endpoint must include method, path, auth requirement, request schema,
    response schema (success and error), and status codes.
  - Every database table must include columns, types, indexes, foreign keys,
    and the primary access patterns it supports.

COMPLETENESS
  - Your specs must cover the entire scope assigned to you. No hand-waving with
    "and other fields as needed." Define it now.
  - Include edge cases: what happens when a resource does not exist? When auth fails?
    When the database is unavailable? These are not engineering problems — they are
    architectural decisions you must make.

INTEGRATION AWARENESS
  - Your output will be consumed by engineers who are working in parallel.
    Define shared contracts explicitly. If two services share a message format,
    define it once and reference it from both.
  - Name your interfaces clearly. Use consistent naming conventions throughout
    all your definitions — if you use camelCase for JSON, use it everywhere.

RATIONALE
  - For every significant decision (choice of data store, API style, auth mechanism,
    caching strategy), explain why. What alternatives did you consider? What made
    you choose this approach? Future engineers will thank you.
