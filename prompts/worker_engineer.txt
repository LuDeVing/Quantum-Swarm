TOOL REFERENCE — what each tool does
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

FILE WRITING (tracked by the project — ONLY these count as "writing a file"):
  write_code_file(filename, content)
      Create or fully replace a source file (Python, JS, TS, etc.).
      filename is relative to the project root, e.g. "app/models.py" or "frontend/src/App.jsx".
      THIS IS THE ONLY TOOL THAT SAVES CODE TO THE PROJECT. Use it for every implementation.

  write_file_section(filename, section_id, content)
      Overwrite a named section inside an existing file without touching the rest.
      Use when you only need to replace one function or class in a large file.

  write_test_file(filename, content)
      Same as write_code_file but saves to the tests/ directory.

  write_config_file(filename, content)
      Save Dockerfile, docker-compose.yml, .env, requirements.txt, package.json,
      nginx.conf, or any other infrastructure/config file to the config/ directory.

⚠ CRITICAL: Using run_shell with printf/echo/cat to create files DOES NOT register the
  file with the project tracking system. The task will be marked as FAILED even if the
  shell command succeeds. Always use write_code_file for code; write_config_file for
  config. run_shell is for commands only.

DISCOVERY (disabled on write-only retry — use at most once per task):
  list_files()
      Lists all files currently in the project. Call this ONCE at the start if you need
      context. Do NOT call it more than twice in a row; you will be penalized.

  read_file(filename)
      Read the current contents of any project file. Use to understand an existing
      implementation before extending or overwriting it.

  search_codebase(query)
      Semantic search for relevant code — e.g. "authentication middleware",
      "SQLAlchemy session", "WebSocket handler". Faster than reading every file.

VALIDATION (always validate after writing):
  validate_python(code)   — checks Python syntax, returns "Python syntax OK" or error.
  validate_json(content)  — validates JSON strings.
  validate_yaml(content)  — validates YAML (Dockerfiles, CI configs, etc.).

EXECUTION (for running tests, installers, build steps — NOT for writing files):
  run_shell(command)
      Runs a shell command in the project's code directory.
      Use for: pip install, npm install, python -m pytest, node script.js, etc.
      Do NOT use to write files (printf/echo/cat redirection will NOT be tracked).

  launch_application(command)
      Starts a desktop app or OS open command in the background (detached). Only works if
      AGENT_LAUNCH_APPS_ENABLED=1. Examples: Windows ``notepad``, ``calc``; macOS ``open -a TextEdit``.
      The model cannot see the window — for web UIs use open_app + browser_action.

  start_service(name, command, port)  — start a background server for testing.
  stop_service(name)                  — stop a background server.
  http_request(method, url, body)     — call an endpoint to verify it works.

BROWSER (for visual smoke-testing a running web UI):
  open_app(url)                          — open browser at a URL.
  browser_action(action, selector, value)— click/type/navigate/screenshot.
  close_browser()                        — ALWAYS release the slot when done.

DESKTOP CONTROL (needs AGENT_DESKTOP_CONTROL_ENABLED=1 — sees and controls your whole screen):
  desktop_screenshot()
      Take a full-screen screenshot and get a Gemini vision description of everything visible.
      Call this before and after every mouse/keyboard action to verify the result.
  desktop_mouse(action, x, y, ...)
      action: 'move' | 'click' | 'double_click' | 'right_click' | 'scroll'
      x, y: pixel coordinates from top-left (-1 = current position)
      scroll_direction: 'up' | 'down'   clicks: scroll ticks
  desktop_keyboard(action, text, keys)
      action: 'type'   → types text character by character
              'hotkey' → holds keys simultaneously  (keys='ctrl,c')
              'press'  → taps keys sequentially     (keys='tab,tab,enter')
  Workflow: desktop_screenshot() → read description → click/type → desktop_screenshot() to verify

TEAM COORDINATION:
  check_dashboard()     — read incoming team messages (call this first in round 2).
  check_messages()      — read direct messages from teammates.
  message_teammate(teammate_role, message) — ask about an interface or warn about a change.
  broadcast_message(message)              — announce a breaking change to everyone.
  request_contract_amendment(file, reason, proposed_change) — request a contract change.
