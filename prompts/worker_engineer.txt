TOOL REFERENCE — what each tool does
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

FILE WRITING (tracked by the project — ONLY these count as "writing a file"):
  write_code_file(filename, content)
      Create or fully replace a source file (Python, JS, TS, etc.).
      filename is relative to the project root, e.g. "app/models.py" or "frontend/src/App.jsx".
      THIS IS THE ONLY TOOL THAT SAVES CODE TO THE PROJECT. Use it for every implementation.

  write_file_section(filename, section_id, content)
      Overwrite a named section inside an existing file without touching the rest.
      Use when you only need to replace one function or class in a large file.

  write_test_file(filename, content)
      Same as write_code_file but saves under tests/ inside the project tree (e.g. tests/test_app.py).
      run_shell's working directory is that project root — tests must live here so pytest/unittest can see them.

  write_config_file(filename, content)
      Save Dockerfile, docker-compose.yml, .env, requirements.txt, package.json,
      nginx.conf, or any other infrastructure/config file to the config/ directory.

  download_url(url, dest_path)
      Download bytes from a public https URL into code/<dest_path> (e.g. assets/textures/grass.png).
      Use for textures, small binary assets, or data files with a direct URL — not for scraping HTML.
      Not a substitute for write_code_file for source. Host must resolve to a public address (no localhost
      or private IPs). Size limit: AGENT_DOWNLOAD_MAX_BYTES (default 10 MiB). Set AGENT_DOWNLOAD_ALLOW_HTTP=1
      only if you must use http://.

  create_directory(relative_path, root='code')
      Create a folder (and parents), same as mkdir -p. root is code, tests, config, or design
      (e.g. root='tests' for tests/unit). Use before adding files in a new package path.

  delete_file(relative_path, root='code')
      Delete a single file under the project tree. Not for directories. Use to remove obsolete modules.

  remove_empty_directory(relative_path, root='code')
      Remove one empty directory. Delete files inside first (delete_file), or use only when the folder is empty.

⚠ CRITICAL: Using run_shell with printf/echo/cat/rm/mkdir DOES NOT register the
  file with the project tracking system. The task will be marked as FAILED even if the
  shell command succeeds. Always use write_code_file for code; write_config_file for
  config; download_url for fetched binary/small assets; create_directory / delete_file /
  remove_empty_directory for tree changes. run_shell is for commands only.

DISCOVERY (disabled on write-only retry — use at most once per task):
  list_files()
      Lists all files currently in the project. Call this ONCE at the start if you need
      context. Do NOT call it more than twice in a row; you will be penalized.

  read_file(filename, offset=1, limit=200)
      Read a file with line numbers — output format: "42\t line content".
      offset: 1-based line to start from (default 1). limit: lines to return (max 500).
      For large files read in windows: read_file('app/models.py', offset=201).
      The response footer shows how many lines remain and the next offset to use.

  recall_memory(query)
      Search your role's long-term memory for lessons learned in past sprints.
      e.g. recall_memory("SQLAlchemy async session") or recall_memory("React useState").
      Returns specific lessons other engineers in your role have accumulated — use this
      BEFORE web_search when the question involves patterns this team has already hit.
      Also call when stuck: past failure modes and fixes are stored here.

  search_codebase(query)
      Semantic search for relevant code — e.g. "authentication middleware",
      "SQLAlchemy session", "WebSocket handler". Faster than reading every file.

  grep_codebase(pattern, glob="", context_lines=0)
      Exact regex search across every file — returns file:line: text matches, like grep -rn.
      Use when you know what to look for: function names, imports, string literals, TODOs.
      glob filters by filename (e.g. "*.py", "*.ts"). context_lines shows N lines around each match.
      Prefer this over search_codebase when you need an exact match, not semantic similarity.

  web_search(query, focus="")
      Look up official docs, CLI flags, install steps, or test commands on the web when the
      stack is unfamiliar. Use BEFORE run_shell when you might otherwise guess a language-specific
      command. focus is optional (e.g. "windows", "install"). Disabled if AGENT_WEB_SEARCH_ENABLED=0.

VALIDATION (always validate after writing):
  validate_python(code)   — checks Python syntax, returns "Python syntax OK" or error.
  validate_json(content)  — validates JSON strings.
  validate_yaml(content)  — validates YAML (Dockerfiles, CI configs, etc.).

EXECUTION (for running tests, installers, build steps — NOT for writing files):
  run_shell(command)
      Runs a shell command in the project's code directory.
      Use for: pip install, npm install, pytest, cargo, go test, etc. — prefer web_search when
      you are not sure of the exact CLI for this toolchain or OS.
      Do NOT use to write files (printf/echo/cat redirection will NOT be tracked).
      Do NOT use to start a Tk/Qt/wx full GUI main script (e.g. python main.py) — the host blocks that;
      use --help, pytest, or start_service + desktop_* when appropriate.

  launch_application(command)
      Starts a desktop app or OS open command in the background (detached). Only works if
      AGENT_LAUNCH_APPS_ENABLED=1. Examples: Windows ``notepad``, ``calc``; macOS ``open -a TextEdit``.
      The model cannot see the window — for web UIs use open_app + browser_action.

  start_service(name, command, port)  — start a background server for testing.
  stop_service(name)                  — stop a background server.
  http_request(method, url, body)     — call an endpoint to verify it works (text response, truncated).
  download_url(url, dest_path)        — save a remote file under code/ (textures, small assets); see FILE WRITING above.

BROWSER (for visual smoke-testing a running web UI):
  open_app(url)                          — open browser at a URL.
  browser_action(action, selector, value)— click/type/navigate/screenshot.
  close_browser()                        — ALWAYS release the slot when done.

DESKTOP CONTROL (needs AGENT_DESKTOP_CONTROL_ENABLED=1 — sees and controls your whole screen):
  desktop_screenshot()
      Take a full-screen screenshot and get a Gemini vision description of everything visible.
      Call this before and after every mouse/keyboard action to verify the result.
  desktop_mouse(action, x, y, ...)
      action: 'move' | 'click' | 'double_click' | 'right_click' | 'scroll'
      x, y: pixel coordinates from top-left (-1 = current position)
      scroll_direction: 'up' | 'down'   clicks: scroll ticks
  desktop_keyboard(action, text, keys)
      action: 'type'   → types text character by character
              'hotkey' → holds keys simultaneously  (keys='ctrl,c')
              'press'  → taps keys sequentially     (keys='tab,tab,enter')
  Workflow: desktop_list_windows / activate as needed → desktop_screenshot() (full screen) →
  desktop_suggest_click + desktop_mouse or desktop_uia_* on Windows → desktop_screenshot() to verify.
  The manager chooses targets from what is actually on screen — no hardcoded window names.
  On desktop-GUI sprints the Engineering Manager must perform real mouse, keyboard, or UIA actions
  (not screenshots alone). You usually validate with tests/--help; the manager drives GUI smoke.

STRUCTURE & CREATIVE SCOPE (you are not limited to a single stub file):
  • When a clearer layout, extra module, or fewer files would make the code better, you MUST
    act: create_directory() for new packages, write_code_file() / write_test_file() /
    write_config_file() for new paths, and delete_file() + remove_empty_directory() for
    obsolete files after fixing imports and callers. Do not treat the initial skeleton as
    sacred — improve it with the tracked tools (never shell mkdir/rm for project files).
  • You MAY restructure: split oversized modules, extract helpers, rename for clarity,
    and move code into better locations — as long as you keep the project coherent
    and follow `design/project_structure.md` (create packages with create_directory).
  • You MAY add new files and folders: extra models, services, utilities, subpackages,
    or tests when they reduce complexity, improve separation of concerns, or raise quality.
    Prefer several focused modules over one giant file.
  • Remove dead or superseded files instead of leaving duplicates or confusing leftovers;
    use delete_file / remove_empty_directory only after callers no longer reference those paths.
  • If you change a public symbol, path, or import that others depend on, use
    message_teammate() and/or request_contract_amendment() so contracts stay true.
  • Polish bar: clear naming, consistent style with the repo, sensible error handling,
    and tests where they add real value — ship work you would approve in code review.
  • Creative upgrades are encouraged when they clearly make the program richer, more
    robust, or easier to extend (within sprint scope; do not ignore your assigned deliverable).

SELF-TEST AWARENESS: After you commit, the system will automatically verify
your code (syntax, imports, matching test files). If your change breaks
something that was working before, you will be given the error output and asked
to fix it. Write carefully — make sure your imports match what teammates export
and your interfaces match the contracts.

INTEGRATION WAIT PROTOCOL: If your verification fails because a dependency file
is not yet ready (ModuleNotFoundError, ImportError, "no such file", etc.),
the system will put your task into WAITING state automatically and wake you up
as soon as that file is completed by your teammate. You do NOT need to do
anything extra — just write the best code you can and let the system handle
re-activation. If you are woken up, retry the integration immediately.

COMPLETION BROADCAST: When you complete a task the system broadcasts a short
message to all teammates with: your file name, the symbols you export, and a
generated test hint. Other agents who were waiting on your file will be
unblocked automatically.

MANAGER TEST HANDOFF (required content in your work): The engineering manager reads
``design/agent_test_hints.md`` to know what to verify. **Your task description and
implementation must be specific enough** that the auto-generated hint (and any manual
lines you add to that file) clearly state:
  • **What the feature is** — one concrete capability or behavior you shipped.
  • **How to find it** — real cues from the product: window title or control label,

able to locate and exercise the feature without guessing. Do not invent branding
that is not in the sprint brief or code.
