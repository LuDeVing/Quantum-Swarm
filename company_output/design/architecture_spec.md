# Architecture Decision Record (ADR-001): Task MVP Foundation

## 1. System Design
*   **Topology:** Containerized micro-services using `docker-compose`.
*   **Network:** API (port 3000), Frontend (port 5173), Postgres (port 5432).
*   **Service Boundaries:** 
    *   `/apps/api`: Fastify, Prisma ORM, TypeScript.
    *   `/apps/web`: React, Vite, TanStack Query (for cache/optimistic state management).
    *   `/packages/contracts`: Shared TypeScript types and `openapi.yaml`.
*   **Data Flow:** Client sends optimistic update via React -> UI updates instantly -> API call initiates -> Reconcile on success or Rollback on error.

## 2. API Contract (Source of Truth)
Base URL: `/api/v1`
All responses return JSON. Dates are strings in ISO8601 format.

| Endpoint | Method | Body | Response (200/201) | Error Codes |
| :--- | :--- | :--- | :--- | :--- |
| `/tasks` | POST | `{title: string}` | `Task` | 400 (Validation), 500 |
| `/tasks` | GET | N/A | `Task[]` | 500 |
| `/tasks/:id` | PUT | `{status: 'PENDING' \| 'COMPLETED'}` | `Task` | 404, 400, 500 |
| `/tasks/:id` | DELETE | N/A | `204 No Content` | 404, 500 |

*   **Auth Header Requirement:** All requests must include `Authorization: Bearer <token>` (Even if ignored by middleware, the header must be present).
*   **Reconciliation:** Every mutation (POST/PUT) must return the full `Task` object.

## 3. Data Model (PostgreSQL)
*   **Table:** `tasks`
    *   `id`: UUID (PK, default `gen_random_uuid()`)
    *   `title`: VARCHAR(100) (NOT NULL)
    *   `status`: ENUM('PENDING', 'COMPLETED') (Default: 'PENDING')
    *   `created_at`: TIMESTAMP WITH TIME ZONE (Default: `NOW()`)
    *   `updated_at`: TIMESTAMP WITH TIME ZONE (Default: `NOW()`)

## 4. Integration Guide (Engineering Execution Path)
1.  **Stage 1 (Contract Setup):** Initialize `packages/contracts`. Export `Task` type and generate `openapi.yaml`. **(Owner: Architecture)**.
2.  **Stage 2 (Database Layer):** Define Prisma schema, run migration. Ensure `docker-compose` spins up the Postgres container. **(Owner: Engineering)**.
3.  **Stage 3 (API Implementation):** Implement routes. Validate input schemas against `packages/contracts`. Ensure 201/200/204 return codes match the contract. **(Owner: Engineering)**.
4.  **Stage 4 (UI/Optimistic Flow):** Implement `useMutation` hooks in React. Integrate the "Rollback" logic. UI must listen for API failure and revert state + trigger notification. **(Owner: Engineering/Design)**.
5.  **Stage 5 (QA Validation):** Run `docker-compose run --rm test`. Verify API adherence to `openapi.yaml`. **(Owner: QA)**.

## 5. Architectural Risks & Mitigation
*   **Risk:** Race conditions during optimistic UI updates.
    *   **Mitigation:** The Frontend MUST use the unique `id` generated by the backend or a client-side temp ID that gets replaced immediately upon arrival of the response.
*   **Risk:** Schema Drift.
    *   **Mitigation:** Any change to `packages/contracts` requires a git-hash increment and a re-run of the CI build.
*   **Bottleneck:** The API-UI reconciliation phase.
    *   **Mitigation:** If the API returns an error on a mutation, the UI must force-refresh the task list from the server to ensure consistency.
