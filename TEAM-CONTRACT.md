## Working Agreements

**Weekly sync — when and where:**
Thursdays at 20:00 via Messenger call (with flexibility to schedule ad-hoc check-ins before Tuesdays if required). 

**Day-to-day communication channel:**
Messenger Group Chat (with a planned migration to a dedicated Discord server as the project scales).

**Code review — how changes get merged:**
For the MVP phase to maximize speed, members are permitted to push directly to the `main` branch. Once the core MVP is stable, we will transition to Pull Requests to ensure code stability.

**What counts as done:**
A task is considered "Done" when:
1. The code is pushed to the repository.
2. It has been tested locally and runs without breaking errors in the console.
3. The UI matches the Figma design.
4. Any new required environment variables have been documented and shared with the team.

Contributions are logged primarily through GitHub commit history. All members agree to write descriptive commit messages (e.g., "feat: added UI for CEO chat" or "fix: resolved Gemini API parsing error"). High-level task tracking will be discussed in our weekly syncs.

## Conflict Resolution

**Technical disagreements — how you decide:**
Disagreements are resolved by a majority vote (2 out of 3). If there is a tie or the vote is unclear, the Tech Lead (Luka) makes the final executive decision regarding the technical architecture.

**If a member misses a deadline:**
Because our roles are highly specialized, a blocked team member halts the whole pipeline. If a member anticipates missing a deadline, they must notify the team at least 24 hours in advance in the chat. The team will then intervene by either: (1) scoping down the feature to make it achievable for the deadline, or (2) scheduling an emergency pair-programming session on a call to help unblock the member. Uncommunicated missed deadlines will result in a team warning.

**Escalation — if the team cannot resolve internally:**
Contact the instructor at zeshan.ahmad@kiu.edu.ge.
